# llm-router — Design

**Status:** Draft v0.2 (counter-proposal)
**Date:** 2026-05-19

This is a leaner alternative to the v0.1 draft. Three changes drive most of it:

1. **No "compile step."** The artifact is a Lua library plus a Lua-table config file. Both are static text; no Go-based templating, no hash-stamped emitter. If you want a hash, hash the files.
2. **Generic first, GenLayer-compatible by emergence.** The router does not bake in determinism, consensus, or any GenLayer-shaped policy. It's a routing library that any Lua-capable host can embed. The properties that make it usable inside GenLayer (privacy-friendly traces, per-process state, configurable profiles) are emergent from sane defaults, not from GenLayer-specific code.
3. **Tooling is optional and minimal.** A couple of Python scripts for validation and benchmarks. No CLI binary in v1.

---

## 1. Scope

### In
- A Lua module (`router.lua`) that, given a `contract`, returns either an LLM response or a structured error, plus a trace.
- A config schema (Lua table) describing providers, models, profiles, scoring, and retry policy.
- A small host interface (HTTP, time, env, log, optional discovery, optional sleep).
- Sync and async (cooperative) execution modes.
- Text, vision, tools, JSON-mode, structured-output passthrough.
- Marketplace discovery for AntSeed-style providers.

### Out (v1)
- Streaming / SSE routing.
- Cost accounting across providers.
- Cross-process state sharing.
- Quality evaluation beyond a static hint per model.
- Any consensus-shaped guarantee. The router does **not** promise that two embedders of the same config will choose the same provider on the same call. Two processes can and will diverge — by design (see §11).

## 2. Architecture

```
       ┌──────────────────────────────┐
       │   config.lua  (Lua table)    │   <- humans write/edit this
       │   metrics.lua (Lua table)    │   <- generated by tools/bench.py
       └──────────────┬───────────────┘
                      │ loaded at host init
                      ▼
       ┌──────────────────────────────┐
       │        router.lua            │
       │   (pure logic + state)       │
       └───────┬──────────────┬───────┘
               │              │
       host.* bindings   public API
               │              │
   ┌───────────┴──┐    ┌──────┴───────────┐
   │ HTTP, time,  │    │ execute,         │
   │ env, log,    │    │ execute_step,    │
   │ discover,    │    │ update_metrics,  │
   │ sleep_ms     │    │ dump/restore     │
   └──────────────┘    └──────────────────┘
```

Three files ship: `router.lua` (the logic), `config.example.lua`, `metrics.example.lua`. The host loads `router.lua`, calls `router.init(config_table, metrics_table)`, installs `host.*` bindings, and then calls `router.execute(contract)` or drives `router.execute_step(...)`.

No emitter, no templating, no hashing built into the library. If a deployer wants integrity guarantees, they hash whatever files they care about with whatever tool they prefer.

## 3. The Contract

The caller's per-call request shape. **Caller-declared**, not config-declared.

```lua
{
  -- content (one of)
  prompt   = "...",
  system   = "...",            -- optional
  messages = { ... },          -- optional, OpenAI shape

  -- multimodal / structured (optional)
  images          = { ... },
  tools           = { ... },
  tool_choice     = "auto",
  response_format = { type = "json_object" },

  -- requirements
  requirements = {
    -- hard filters: candidates that fail are eliminated entirely
    needs        = { "tools", "json_mode" },        -- capability flags required
    min_context  = 32000,                           -- minimum context window
    min_tok_s    = 30,                              -- minimum observed throughput (from metrics.lua)
    min_quality  = 0.7,                             -- minimum static_quality_hint
    privacy      = "standard",                      -- "standard" | "no_log" | "tee_required"
    tier         = "partner",                       -- "partner" | "marketplace" | "fallback" (allow-only)
    model_family = "claude-opus-4.7",               -- restrict to one model family
    pin          = { provider = "comput3",          -- bypass routing; route to this tuple only
                     model    = "hermes-3-405b" },

    -- soft targets: candidate that exceeds these gets a scoring penalty, not eliminated
    max_latency_ms = 5000,
    max_cost_usd   = 0.01,
  },

  -- selection
  profile = "default",         -- selects a named profile in config

  -- per-call weight override (mixed with profile weights)
  weights_override = { cost = 1.0, quality = 0, partner = 0 },

  -- debugging
  trace = true,                -- include trace in response (host should not publish it)
}
```

Hard requirements filter the candidate set. Soft requirements affect scoring. Privacy levels filter by provider capability flags (`no_log`, `has_tee`).

`weights_override` is merged onto the resolved profile weights at scoring time. Keys present in the override fully replace the corresponding profile weight; absent keys fall through to the profile. After merge, weights are renormalized so the max achievable score is 1.0. Use this for one-off ad-hoc requests without authoring a whole profile (e.g. "for this one call, only cost matters").

`requirements.pin` short-circuits routing: filtering, scoring, and the fallback chain are all skipped. The router calls `host.call_provider(pin.provider, pin.model, ...)` once and returns. If the call fails, the router does not try anything else. This is the escape hatch for "I know exactly what I want; don't make decisions for me."

**Deliberately removed from v0.1:** `determinism`, `pin_quantization`, `pin_seed`. These belong in the caller's domain (or the consumer's policy), not the router's. The `pin` field above replaces the use case of "I need a specific provider+model" without the consensus-shaped framing of v0.1. See §11.

## 4. Profiles

Named bundles of `(weights, retry_policy, hard_constraints, fallback_strategy)`. The config file declares profiles; the contract selects one by name. Profiles support inheritance for ergonomic reuse, resolved at `router.init` time.

```lua
profiles = {
  default = {
    weights = { quality = 0.4, speed = 0.3, cost = 0.2, free_credit = 0.05, partner = 0.05 },
    retry_policy = "balanced",     -- references a named retry table
    hard_constraints = {},
    fallback_strategy = "score_order",
  },

  cheap_explore = {
    extends = "default",
    weights = { quality = 0.2, speed = 0.2, cost = 0.5, free_credit = 0.1 },
  },

  tee_only = {
    extends = "default",
    hard_constraints = { privacy = "tee_required" },
    weights = { quality = 0.5, speed = 0.3, cost = 0.2 },
  },
}
```

The router ships **no** built-in profiles. The example config has a few. Consumers are expected to author their own.

## 5. Host Interface

The host provides these globals to the Lua VM. The router never does I/O itself.

```lua
-- Required
host.call_provider(provider_id, model_id, request, options) -> {
  ok, status, body, raw_body, latency_ms, error_kind, error_detail
}
host.now_ms() -> number
host.env(name) -> string | nil
host.log(level, event, fields)      -- structured event; host decides destination

-- Optional
host.discover(discovery_id) -> { ok, offers | nil, error | nil }   -- for marketplace
host.sleep_ms(ms)                   -- sync mode only
host.persist_state(key, value)      -- if host wants warm-start state across restarts
host.load_state(key) -> value | nil
```

`host.log` is intentionally **one** function, not two. The router emits structured events; the host decides what's local, what's published, what's discarded. The router has no opinion about disclosure.

### Canonical error kinds

The host classifies provider responses into a fixed set: `ok`, `rate_limit`, `timeout`, `server_error`, `auth_error`, `bad_request`, `content_filter`, `model_unavailable`, `context_overflow`, `network_error`, `unknown`. Each kind has a default action in the retry table (§7).

### Sync vs async

`router.execute(contract)` blocks the Lua thread until terminal success/failure. Uses `host.sleep_ms` if present.

`router.execute_step(state_handle, contract_or_nil) -> step` is the cooperative form. Host calls it in a loop. Returns either `{ status = "done", result = ... }` or `{ status = "wait", until_ms = ..., state_handle = ... }`. First call passes the contract; subsequent calls pass `nil` (state is carried through `state_handle`).

GenVM and any wasmtime host uses `execute_step`. Plain Python scripts use `execute`.

## 7. Scoring and Retry

### Score

For each `(provider, model)` that survives hard filtering:

```
score = w_quality * Q + w_speed * S + w_cost * C
      + w_free * free_indicator + w_partner * partner_score
      - penalty_circuit_breaker(provider)
```

`Q`, `S`, `C` are normalized to `[0,1]`. The normalizations are explicit and documented:

- `Q = static_quality_hint` (a flat scalar in v1; per-task quality is future work).
- `S = clamp(1 - ema_latency_ms / max_latency_ms_target, 0, 1)` where `max_latency_ms_target` is per-profile.
- `C = clamp(1 - effective_cost_per_call / max_cost_usd, 0, 1)` where `effective_cost_per_call` is estimated from the catalog's price + expected output tokens (a coarse heuristic, configurable).
- `free_indicator` is `1` if the provider has free credits available above a threshold, else `0`. Determined from `metrics.lua` (it's static between bench runs; not a live counter).
- `partner_score`: `1` for `tier=partner`, `0.5` for `marketplace`, `0` for `fallback`.

Weights are renormalized at `init` time so the max achievable score is 1.0 per profile. This is internal and shouldn't surprise the user — but documenting it explicitly avoids "why is my w_cost=10 not making cost dominate?" confusion.

### Retry

A named retry policy maps error kinds to actions:

| Error kind | Default action |
|---|---|
| `ok` | (terminal) |
| `rate_limit` | `next_candidate` + open breaker (default 30s) |
| `timeout` | `next_candidate` + EMA latency penalty |
| `server_error` | `retry_same` ×1 (500ms backoff), then `next_candidate` |
| `auth_error` | `disable_provider` for the rest of the process |
| `bad_request` | `abort` |
| `content_filter` | `next_candidate` or `abort` (profile choice) |
| `model_unavailable` | `next_provider_same_model`, mark unavailable for 5min |
| `context_overflow` | `abort` |
| `network_error` | `retry_same` ×2 (200ms/600ms), then `next_candidate` |
| `unknown` | `next_candidate` |

Actions: `retry_same`, `next_candidate`, `next_provider_same_model`, `disable_provider`, `abort`.

### Circuit breaker

Per-provider, in mutable state. Opens after N consecutive non-`ok` outcomes or on explicit `rate_limit`. While open, score = 0. Default N=3, default open duration 30s for rate limit, 5min for repeated failure.

## 8. Trace and State

### Trace

`router.execute` returns `{ ok, response, error, trace, chosen }`. `trace` is a list of `AttemptRecord`:

```lua
{
  provider, model, served_model_id, attempt_n,
  score, score_breakdown,
  started_at_ms, duration_ms,
  outcome,            -- error_kind or "ok"
  action_taken,
  notes,
}
```

The trace is **local-only by default**. The router emits it back to the caller. The host decides whether to log it, persist it, publish it, or discard it. The router itself never publishes anything to a network destination.

### State

Mutable state lives in the Lua module's runtime, per process:

```lua
{
  circuit_breakers     = { [provider] = { open, opened_at_ms, consecutive_failures } },
  ema_metrics          = { [pm_key]  = { ema_latency_ms, success_rate_ewma, n } },
  disabled_providers   = { [provider] = reason_string },
  discovery_cache      = { [discovery_id] = { offers, fetched_at_ms } },
}
```

`router.dump_state()` returns a serializable copy. `router.restore_state(snapshot)` restores it. The host can persist between restarts via `host.persist_state` if it wants warm starts; otherwise everything is rebuilt at boot.

State is **not** designed to be shared between processes. Two routers running side-by-side will, over time, develop different circuit breaker states, different EMA latencies, and may make different choices on the same call. This is fine — it's emergent diversity, not a correctness problem.

### `metrics.lua` vs runtime EMA — two distinct things

It is easy to conflate `metrics.lua` (a file) with EMA state (in memory). They serve different roles:

| | `metrics.lua` | Runtime EMA state |
|---|---|---|
| Source | Generated by `tools/bench.py` | Accumulated during process lifetime |
| When loaded | Once at `router.init` | Initialized from `metrics.lua` then updated per call |
| Persistence | File on disk; survives restart | In-memory; lost on restart unless `dump_state`/`restore_state` is used |
| Granularity | Snapshot from last bench run | Live observations from this process |
| Used for | Cold-start scoring (no observations yet) | Warm scoring (after N>5 observations, EMA dominates) |

`router.update_metrics(provider_id, model_family, delta)` lets the host push observations *into* the runtime EMA without going through `host.call_provider` — useful for sidecars that observe usage out of band (e.g. a credits-monitoring process).

## 9. Marketplace Discovery (AntSeed)

The catalog declares AntSeed with `discovery = "marketplace"` and no static models. At scoring time, when AntSeed is in the candidate set, the router calls `host.discover("antseed_buyer_node")`. Offers are merged into the candidate set with `provider_id = "antseed"`; they score like any other candidate.

### Offer schema

```lua
host.discover("antseed_buyer_node") -> {
  ok = true,
  fetched_at_ms = ...,            -- echoed by router into state cache
  offers = {
    {
      model_family    = "llama-3.3-70b",
      seller_endpoint = "https://seller-foo.antseed.com/v1",
      price_in_usd_per_mtok  = 0.10,
      price_out_usd_per_mtok = 0.30,
      est_tok_s       = 38,
      capabilities    = { supports_tools = true, context = 128000, ... },
      quote_id        = "qid_abc123",        -- optional, opaque to router
      expires_at_ms   = 1716127200000,       -- optional; expired offers are skipped
    },
    ...
  },
}
```

The router treats each offer as a candidate with `provider_id="antseed"` and `model = offer.model_family`. When the router selects an offer, it passes the entire offer table through to `host.call_provider("antseed", model, request, options)` inside `options.offer`. The host uses `seller_endpoint` as the base URL and includes `quote_id` / any other protocol-specific fields when dispatching the HTTP request.

### Three host implementation paths

The router doesn't care how the host produces offers. Three reasonable implementations:

1. **Scrape `antseed.com/network`** — zero infra, fragile to HTML changes. Smoke test only.
2. **Query `antseed.com/docs/pricing` JSON API** — zero infra, lightly cached on AntSeed's side. **Default sensible choice for most consumers.**
3. **Local buyer node sidecar** — live pricing, full marketplace participant, requires running a process. **Production path for serious validators.**

The same `host.discover` signature serves all three.

### Caching

Results are cached in mutable state for a configurable TTL (default 60s) to avoid hammering the discovery source. The host can flush via `router.invalidate_discovery("antseed")`.

### Out of scope for the router

- **Payments and settlement.** Provider-side and host-side concern.
- **Per-seller reputation.** v1 circuit breaker granularity is per `provider_id`, so all AntSeed sellers share one breaker. v2 may add per-`seller_endpoint` breakers if needed.
- **Quote expiry between discovery and dispatch.** If the seller rejects an expired quote, the host classifies as `bad_request` (or returns `unknown`) and the router moves to the next candidate.

## 10. Multimodal, Tools, JSON Mode

Capability-driven, no special code paths:

- `images` non-empty → `vision` added to `needs` → filter by `supports_vision`.
- `tools` non-empty → `tools` added to `needs` → filter by `supports_tools`. `tool_choice` is passthrough.
- `response_format = { type = "json_object" }` → `json_mode` added → filter by `supports_json_mode`.

The host's `call_provider` is responsible for translating to provider-specific request shapes when they deviate from OpenAI conventions (e.g., Anthropic, Google). The router speaks OpenAI shape natively.

## 11. Notes on Private / Multi-Validator Deployments

This section is informational. The router itself does not enforce any of it.

The router supports use cases where **each embedder is expected to keep its config private**: which providers it prefers, what weights it uses, what its API keys are, what its current circuit breaker state is. The library helps this case in a few small ways:

- **No shared state.** Two routers don't synchronize. Their state diverges naturally.
- **No built-in profiles.** The library ships templates, not defaults. Two operators starting from the same template are mildly nudged to diverge through their own tuning.
- **Local-only trace.** The router never publishes traces. The host can choose to log them locally for forensics (e.g. for appeals in a GenLayer-style consensus context) without exposing them on a wire.
- **Single `log` channel.** The host owns disclosure policy. The router has no concept of "this event is public" vs "this event is private."

For a GenLayer validator: this means the router is usable, but the validator (not the router) decides what to publish, what to keep, and how to vary its config relative to other validators. The router does **not** claim to make consensus easier. It does claim to make running a privately-configured LLM frontend easier.

A separate threat-model document (forthcoming) should validate these properties against the actual greyboxing requirements before this section is treated as a feature spec.

## 12. Tooling

Two Python scripts in `tools/`. No CLI binary.

### `tools/validate.py`
- Loads `config.lua` and `metrics.lua` via lupa.
- Checks schema (required fields, valid enums, references resolve).
- Warns about referenced `auth_env` variables that are unset in the current environment.
- Exits non-zero on schema errors.

### `tools/bench.py`
- Loads `config.lua`.
- Runs N canned prompts (default: 5, ~500 tokens each, mix of short reasoning + JSON + tool-call) against each enabled provider/model.
- Measures: end-to-end latency, output tok/s, success rate.
- Writes a `metrics.lua` file.
- Single process, sequential, no parallelism (avoid triggering rate limits during bench).

Both scripts are ~200 lines each. They're convenience, not core. A deployer who doesn't want them can write their own.

## 13. Operating

Once a router is deployed, four maintenance tasks recur. They have different frequencies and owners; conflating them is a recipe for confusion.

| Task | Frequency | Owner | Where it lives |
|---|---|---|---|
| 1. Add/remove providers and models | When partnerships or model availability change | Deployer | `config.lua` |
| 2. Refresh observed metrics | Periodic (nightly/weekly cron) | Automated via `tools/bench.py` | `metrics.lua` |
| 3. Maintain `host.call_provider` adapters | When a provider changes their API surface | Host author | Host code (not the router) |
| 4. Verify catalog capability flags | Continuous (ideally) | Bench harness — partial in v1 | `metrics.lua` capability checks |

**Task 1** is rare and manual.

**Task 2** is the obvious one and the only one fully automated in v1. If skipped, `metrics.lua` ages and the cold-start scoring uses stale data; the runtime EMA still corrects over time per process, but newly started processes get worse decisions for their first N calls.

**Task 3** is the silent maintenance burden most consumers underestimate. Provider APIs drift — error response shapes change, rate-limit markers change, new HTTP status codes appear. The `host.call_provider` adapter is what classifies responses into canonical `error_kind` values, and when that classification drifts (e.g. a real rate-limit gets classified as `unknown`), the circuit breaker no longer opens correctly and you end up hammering a broken provider. This code lives **per host**, not in `router.lua`, and needs its own tests.

**Task 4** is the hardest because failure is silent. A model's `supports_tools = true` flag is an assertion that can become false (model deprecated, provider stopped serving it, provider serves a quantized version that breaks tool-calling). v1 has no formal verification path; `tools/bench.py` checks latency and success but not capability fidelity. v2 should add a capability-check pass that compares declared flags against observed behavior and emits warnings.

## 15. Public API

```lua
router.init(config_table, metrics_table) -> ok | error

router.execute(contract) -> {
  ok, response, error, trace, chosen
}

router.execute_step(state_handle, contract_or_nil) -> {
  status = "done" | "wait",
  result = ... ,           -- when done
  until_ms, state_handle,  -- when wait
}

router.update_metrics(provider_id, model_family, delta)
router.invalidate_discovery(discovery_id)

router.dump_state() -> snapshot
router.restore_state(snapshot)

router.info() -> { version, profile_names, providers_loaded, models_loaded }
```

No `M.compile_hash`, no `M.config_hash`. If a deployer needs reproducibility guarantees, they hash whichever files they care about externally and stamp the result wherever they want.
